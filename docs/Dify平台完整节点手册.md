## 节点分类概览

Dify 平台提供了丰富的节点库，可分为以下几大类：

| 分类 | 节点数量 | 主要用途 |
|------|--------|--------|
| **流程控制** | 3个 | 定义工作流的起点、终点和回复方式 |
| **LLM & AI** | 4个 | 调用大模型、知识检索、意图识别、参数提取 |
| **数据处理** | 3个 | 数据转换、代码执行、模板渲染 |
| **逻辑控制** | 4个 | 条件判断、分支合并、循环迭代、重复执行 |
| **外部集成** | 2个 | HTTP 请求、调用工具 |
| **工作流组织** | 1个 | 变量赋值 |

**总计：17 个核心节点**

## 流程控制节点

### 1. 开始节点（Start）

**图标**：🟢

**功能说明**：
- 定义工作流的入口点和初始输入参数
- 设置工作流的输入变量和类型
- 所有工作流必须包含至少一个开始节点

**配置参数**：
```
输入变量名：string / number / array / object / boolean
变量描述：选填，用于说明该变量的含义
是否必填：是/否
```

**使用场景**：
- 接收用户输入
- 接收 API 请求参数
- 定义流程的初始数据源

**示例配置**：
```
变量名: query
类型: string
描述: 用户问题
必填: 是

变量名: user_id
类型: string
描述: 用户唯一标识
必填: 否
```

---

### 2. 结束节点（End）

**图标**：🔴

**功能说明**：
- 定义工作流的出口点和最终输出内容
- 工作流执行结果返回给调用方
- Workflow 模式的最终输出（区别于 Chatflow）

**配置参数**：
```
输出变量：从上游节点选择要输出的变量
```

**使用场景**：
- 定义工作流的返回结果
- 返回结构化的输出数据
- 在自动化工作流中标记流程完成

**示例配置**：
```
{
  "outputs": {
    "answer": "最终答案内容",
    "sources": "参考来源"
  }
}
```

---

### 3. 回复节点（Answer）

**图标**：💬

**功能说明**：
- 在 **Chatflow**（对话流）模式中定义回复内容
- 用于定义 AI 助手的直接回复
- 支持动态变量插值和 Markdown 格式

**配置参数**：
```
回复内容：支持变量引用 {{variable_name}}
回复格式：纯文本 / Markdown / HTML
```

**使用场景**：
- 在对话式应用中回复用户
- 支持流式输出（streaming）
- 与 Chatbot 应用配合使用

**示例配置**：
```
根据您的问题 {{query}}，
我从知识库找到的答案是：
{{knowledge_result}}

参考文献：{{source_citation}}
```

## LLM 与 AI 节点

### 4. 大语言模型节点（LLM）

**图标**：🤖

**功能说明**：
- 调用大语言模型进行文本处理、推理、生成等任务
- 支持多种 LLM 模型（GPT-4、Claude、Qwen、Llama 等）
- 支持动态模型切换、异步推理、结果校验

**核心配置参数**：

| 参数 | 说明 | 示例值 |
|------|------|-------|
| **模型选择** | 选择调用的 LLM 模型 | GPT-4 / Claude 3.5 |
| **系统提示词** | LLM 的角色和指令 | "你是一个专业的数据分析师" |
| **用户消息** | 传递给 LLM 的主要内容 | {{user_query}} |
| **温度值(Temperature)** | 控制生成随机性(0-2) | 0.7（通常在0.3-0.8） |
| **最大输出长度** | 限制生成的 token 数 | 1000 |
| **Top-P** | 核心采样参数(0-1) | 0.9 |
| **Stop 词** | 生成停止词 | ```\n``` |

**功能特性**：
- ✅ 支持多轮对话上下文管理
- ✅ 支持 Function Calling（工具调用）
- ✅ 支持模型微调和自定义配置
- ✅ 支持结构化输出（JSON Schema）

**使用场景**：
- 问答系统
- 内容生成（文案、文章、代码）
- 数据分析和摘要
- 翻译和代码优化
- 对话机器人

**提示词示例**：
```
系统提示词：
你是一个友好的客服助手，专门帮助用户解决产品问题。
- 使用简洁、专业的语言
- 如果不知道答案，主动提出转接人工客服
- 始终保持礼貌和耐心

用户消息模板：
用户问题：{{query}}
上下文信息：{{context}}
相关商品信息：{{product_info}}
```

### 5. 知识检索节点（Knowledge Retrieval）

**图标**：📚

**功能说明**：
- 从 Dify 知识库中检索与用户问题相关的文本片段
- 基于向量相似度搜索，返回最匹配的文档内容
- 可作为下游 LLM 节点的上下文信息

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **知识库选择** | 选择要检索的目标知识库 |
| **查询文本** | 用于搜索的查询内容（通常是用户问题） |
| **返回结果数量** | 返回几条最相关的文档片段（通常1-5条） |
| **相似度阈值** | 相似度分数的最低要求（0.5-0.8） |
| **自定义顶部提示** | 是否使用针对该知识库的特殊检索指令 |

**返回数据结构**：
```json
{
  "results": [
    {
      "content": "检索到的文档片段",
      "source": "文档来源",
      "similarity_score": 0.92,
      "metadata": {
        "document_name": "文件名",
        "page": 3
      }
    }
  ],
  "retrieval_used": true
}
```

**使用场景**：
- ✅ RAG（Retrieval-Augmented Generation）系统核心
- ✅ 知识库问答（FAQ、文档查询）
- ✅ 企业级搜索引擎
- ✅ 帮助文档智能查询

**最佳实践**：
```
1. 在 LLM 节点前执行知识检索
2. 将检索结果作为 Context 传入 LLM 的系统提示词
3. 示例提示词：
   "基于以下参考资料回答问题：
   {{retrieval_results}}
   
   用户问题：{{query}}"
```

### 6. 问题分类节点（Question Classifier）

**图标**：🏷️

**功能说明**：
- 通过定义分类规则，让 LLM 自动分类用户输入
- 将用户问题自动分配到对应的业务分类
- 支持多分类或多标签分类

**核心配置参数**：

| 参数 | 说明 | 示例 |
|------|------|------|
| **输入问题** | 要分类的用户问题 | {{user_query}} |
| **分类选项** | 定义所有可选的分类 | ["销售", "售后", "投诉", "咨询"] |
| **分类描述** | 每个分类的详细说明 | "销售: 关于产品购买、价格、优惠的问题" |

**输出结果示例**：
```json
{
  "category": "售后",
  "confidence": 0.95,
  "reason": "用户询问退货相关问题，属于售后类别"
}
```

**使用场景**：
- 🎯 智能客服分类（销售/售后/投诉/建议）
- 🎯 内容分类和标签
- 🎯 自动路由到不同工作流
- 🎯 意图识别系统

**配置示例**：
```
输入: "我的订单什么时候发货？"

分类选项:
1. 销售/咨询 - 关于购买、价格、产品信息
2. 订单/物流 - 关于发货、配送、订单查询
3. 退货/售后 - 关于退换货、维修、退款
4. 投诉/反馈 - 关于投诉、建议、意见

输出: 
{
  "category": "订单/物流",
  "confidence": 0.98
}
```

### 7. 参数提取节点（Parameter Extractor）

**图标**：🔍

**功能说明**：
- 利用 LLM 从自然语言中提取结构化的参数
- 将非结构化文本转换为结构化数据
- 用于后续的工具调用或 HTTP 请求

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **输入文本** | 需要提取参数的自然语言文本 |
| **参数定义** | 定义要提取的参数名、类型、描述、是否必填 |
| **提取指令** | LLM 的提取规则说明 |

**参数定义结构**：
```json
{
  "parameters": [
    {
      "name": "destination",
      "type": "string",
      "description": "目的地城市名称",
      "required": true
    },
    {
      "name": "travel_date",
      "type": "string",
      "description": "出发日期（格式：YYYY-MM-DD）",
      "required": true
    },
    {
      "name": "number_of_people",
      "type": "integer",
      "description": "旅行人数",
      "required": false,
      "default": 1
    }
  ]
}
```

**提取结果示例**：
```
用户输入: "我要订三张去北京的机票，下周一出发"

提取结果:
{
  "destination": "北京",
  "travel_date": "2026-04-07",
  "number_of_people": 3
}
```

**使用场景**：
- 🔧 API 接口参数智能生成
- 🔧 表单自动填充
- 🔧 订单系统的自然语言下单
- 🔧 预订系统（机票、酒店、餐厅）

**工作流示例**：
```
用户输入消息
    ↓
参数提取节点 (提取城市、日期、人数)
    ↓
HTTP 请求节点 (调用订票 API)
    ↓
返回订票结果
```

## 数据处理节点

### 8. 代码执行节点（Code）

**图标**：💻

**功能说明**：
- 在工作流中执行 Python 或 Node.js 代码
- 实现复杂的数据转换、计算、逻辑处理
- 支持自定义库和模块导入

**支持语言**：
- 🐍 Python 3.9+
- 📘 Node.js 16+

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **语言选择** | Python 或 JavaScript |
| **代码编辑区** | 编写自定义代码 |
| **输入变量** | 从上游节点接收的变量 |
| **代码逻辑** | 数据处理逻辑 |

**Python 示例**：
```python
# 输入变量：input_text, numbers_list
import json
import math

# 数据处理
cleaned_text = input_text.strip().lower()
sum_of_numbers = sum(numbers_list)
average = sum_of_numbers / len(numbers_list)
max_value = max(numbers_list)

# 构建输出
result = {
    "cleaned_text": cleaned_text,
    "statistics": {
        "sum": sum_of_numbers,
        "average": round(average, 2),
        "max": max_value,
        "count": len(numbers_list)
    }
}

# 必须返回字典格式的输出
# output 变量名会被自动识别
output = json.dumps(result, ensure_ascii=False)
```

**JavaScript 示例**：
```javascript
// 输入变量：order_items, discount_rate
const calculateTotal = (items, discount) => {
  const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const discountAmount = subtotal * discount;
  const total = subtotal - discountAmount;
  
  return {
    subtotal: subtotal.toFixed(2),
    discount_amount: discountAmount.toFixed(2),
    total: total.toFixed(2),
    item_count: items.length
  };
};

const result = calculateTotal(order_items, discount_rate);
// 输出需要是对象格式
result;
```

**可调用的内置库**：

| 库名 | 用途 |
|------|------|
| **json** / **JSON** | JSON 解析和序列化 |
| **math** | 数学运算（Python） |
| **datetime** | 日期时间处理（Python） |
| **requests** | HTTP 请求（Python） |
| **hashlib** | 哈希计算（Python） |
| **re** | 正则表达式（Python/JS） |

**使用场景**：
- 📊 复杂数据计算和转换
- 📊 字符串处理和解析
- 📊 条件逻辑实现
- 📊 数据清洗和格式化
- 📊 与外部 Python/Node 库的集成

**常见用途**：
```python
# 1. 数据清洗
output = {
    "cleaned": [x.strip().lower() for x in data_list if x],
    "item_count": len(data_list)
}

# 2. 计算和统计
output = {
    "total": sum(numbers),
    "average": sum(numbers) / len(numbers),
    "min": min(numbers),
    "max": max(numbers)
}

# 3. 条件分支
if sales_amount > 10000:
    output = {"level": "VIP", "discount": 0.2}
else:
    output = {"level": "普通", "discount": 0.05}
```

### 9. 模板转换节点（Template）

**图标**：📄

**功能说明**：
- 使用 Jinja2 Python 模板语言进行灵活的数据转换
- 支持条件判断、循环、过滤等模板语法
- 用于生成动态文本、HTML、JSON 等格式

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **模板内容** | 使用 Jinja2 语法编写的模板 |
| **输入变量** | 传入模板的数据变量 |

**Jinja2 语法示例**：

```jinja2
{# 基础变量替换 #}
用户名：{{ user_name }}
邮箱：{{ email }}

{# 条件语句 #}
{% if user_level == 'VIP' %}
  尊敬的 VIP 用户 {{ user_name }}
{% elif user_level == '普通' %}
  亲爱的用户 {{ user_name }}
{% else %}
  欢迎新用户
{% endif %}

{# 循环语句 #}
订单列表：
{% for order in orders %}
  - 订单号：{{ order.id }}
    产品：{{ order.product }}
    金额：¥{{ order.amount }}
{% endfor %}

{# 过滤器 #}
标题：{{ title | upper }}
日期：{{ create_time | strftime('%Y-%m-%d') }}
金额：{{ price | round(2) }}

{# 数组/对象访问 #}
第一个产品：{{ products[0].name }}
用户地址：{{ user_info.address.city }}

{# 三元表达式 #}
状态：{{ '已完成' if status == 'done' else '进行中' }}

{# 长度和计数 #}
总共有 {{ items | length }} 个项目

{# 字符串连接 #}
完整信息：{{ first_name }} {{ last_name }} - {{ company }}
```

**常用 Jinja2 过滤器**：

| 过滤器 | 说明 | 示例 |
|-------|------|------|
| `upper` | 转大写 | `{{ text \| upper }}` |
| `lower` | 转小写 | `{{ text \| lower }}` |
| `capitalize` | 首字母大写 | `{{ text \| capitalize }}` |
| `length` | 获取长度 | `{{ items \| length }}` |
| `join` | 数组连接 | `{{ names \| join(', ') }}` |
| `split` | 字符串分割 | `{{ text \| split(',') }}` |
| `round` | 四舍五入 | `{{ price \| round(2) }}` |
| `default` | 默认值 | `{{ value \| default('N/A') }}` |
| `abs` | 绝对值 | `{{ number \| abs }}` |

**使用场景**：
- 📝 生成格式化的邮件、报告内容
- 📝 HTML / JSON 动态生成
- 📝 提示词的动态构建
- 📝 复杂的文本格式化

**实践示例**：

```jinja2
{# 邮件模板 #}
尊敬的 {{ customer_name }}，

感谢您的订单！订单详情如下：

订单号：{{ order_id }}
下单时间：{{ order_date | strftime('%Y年%m月%d日') }}
订单金额：¥{{ order_total | round(2) }}

订单商品：
{% for item in order_items %}
  {{ loop.index }}. {{ item.name }}
     数量：{{ item.quantity }}
     单价：¥{{ item.price }}
     小计：¥{{ item.price * item.quantity | round(2) }}
{% endfor %}

订单状态：
{% if order_status == 'pending' %}
  ⏳ 待支付
{% elif order_status == 'shipped' %}
  📦 已发货，快递单号：{{ tracking_number }}
{% elif order_status == 'delivered' %}
  ✅ 已送达
{% endif %}

预计送达：{{ estimated_delivery_date }}

{% if order_total > 500 %}
🎁 您获得了 10% 的优惠券
{% endif %}

有任何问题欢迎咨询客服。

祝购物愉快！
```

## 逻辑控制节点

### 11. 条件分支节点（IF/ELSE）

**图标**：❓

**功能说明**：
- 根据条件将工作流拆分成两个分支（真分支 / 假分支）
- 支持复杂的条件组合（AND / OR / NOT）
- 基于上游节点的输出变量进行条件判断

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **条件表达式** | 评估的条件（支持逻辑运算） |
| **真分支路径** | 条件成立时执行的节点链 |
| **假分支路径** | 条件不成立时执行的节点链 |

**支持的条件操作符**：

| 运算符 | 说明 | 示例 |
|-------|------|------|
| `==` | 等于 | `user_level == 'VIP'` |
| `!=` | 不等于 | `status != 'completed'` |
| `>` | 大于 | `order_amount > 1000` |
| `>=` | 大于等于 | `age >= 18` |
| `<` | 小于 | `stock < 10` |
| `<=` | 小于等于 | `discount <= 0.2` |
| `in` | 包含 | `category in ['电子', '图书']` |
| `contains` | 字符串包含 | `text contains '紧急'` |
| `is empty` | 为空 | `reason is empty` |
| `is not empty` | 不为空 | `feedback is not empty` |

**逻辑组合**：
```
条件A AND 条件B    # 同时满足
条件A OR 条件B     # 满足其中一个
NOT 条件A          # 不满足该条件
(条件A OR 条件B) AND 条件C   # 复杂组合
```

**使用场景**：
- 🌳 VIP 用户不同处理流程
- 🌳 订单金额判断（是否需要审批）
- 🌳 库存判断（是否可下单）
- 🌳 优先级路由（紧急/普通/低优先级）

**实例工作流**：
```
用户输入订单
    ↓
检查订单金额
    ↓
条件分支：订单金额 > 10000?
    ├─ 是 → 需要审批流程 → 发送审批邮件
    └─ 否 → 直接处理 → 生成订单确认
    ↓
回复用户
```

### 12. 循环节点（Loop）

**图标**：🔄

**功能说明**：
- 对列表对象执行多次步骤直至遍历完所有项
- 每次迭代都会处理列表中的一项
- 支持设置最大循环次数和退出条件

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **循环列表** | 要迭代的数组/集合 |
| **循环节点** | 每次迭代执行的步骤 |
| **最大循环次数** | 防止无限循环 |
| **循环变量** | 当前迭代项（如 item） |

**使用场景**：
- 🔁 批量处理数据
- 🔁 对列表中的每个元素进行相同操作
- 🔁 生成批量结果

**示例工作流**：
```
输入: 产品列表 [商品1, 商品2, 商品3]

循环处理：
├─ 第1次 → 检查商品1库存 → 调用LLM生成描述 → 保存结果
├─ 第2次 → 检查商品2库存 → 调用LLM生成描述 → 保存结果
├─ 第3次 → 检查商品3库存 → 调用LLM生成描述 → 保存结果

输出: 3条商品的完整信息
```

### 13. 迭代节点（Iteration）

**图标**：🔁

**功能说明**：
- 执行依赖前一轮结果的重复任务
- 直到满足退出条件或达到最大迭代次数
- 用于逐步演进结果（如多轮对话、递进式生成）

**核心配置参数**：

| 参数 | 说明 |
|------|------|
| **初始值** | 迭代的起点 |
| **迭代步骤** | 每轮执行的操作 |
| **退出条件** | 何时停止迭代 |
| **最大迭代次数** | 防止无限循环 |

**使用场景**：
- 📈 多轮对话优化
- 📈 递进式内容生成（大纲 → 段落 → 完整文章）
- 📈 根据反馈的反复修改
- 📈 知识图谱逐步构建

**示例工作流**：
```
用户提供写作主题
    ↓
第1轮迭代：生成文章大纲
    ↓
第2轮迭代：根据大纲扩展各章节
    ↓
第3轮迭代：检查质量，修改不满意的部分
    ↓
满足质量要求？
    ├─ 否 → 继续迭代
    └─ 是 → 输出最终文章
```

## 外部集成节点

### 14. HTTP 请求节点（HTTP Request）

**图标**：🌐

**功能说明**：
- 通过 HTTP 协议发送服务器请求
- 调用第三方 API、Webhook、生成图片等
- 支持 GET、POST、PUT、DELETE 等 HTTP 方法

**核心配置参数**：

| 参数 | 说明 | 示例 |
|------|------|------|
| **URL** | 请求的目标地址 | `https://api.example.com/users` |
| **方法** | HTTP 方法 | GET / POST / PUT / DELETE |
| **请求头** | HTTP Headers | `Authorization: Bearer token` |
| **查询参数** | URL 参数（GET） | `?user_id=123&page=1` |
| **请求体** | JSON Body（POST/PUT） | `{"name": "张三"}` |
| **超时时间** | 请求超时秒数 | 30 |
| **重试次数** | 失败重试次数 | 3 |

**常见 HTTP 请求示例**：

```
# GET 请求 - 查询天气
GET https://api.weatherapi.com/v1/current.json?q=Beijing

# POST 请求 - 创建用户
POST https://api.example.com/users
Headers: Content-Type: application/json
Body: {"name": "李四", "email": "li@example.com"}

# PUT 请求 - 更新订单
PUT https://api.example.com/orders/12345
Headers: Authorization: Bearer xxx
Body: {"status": "shipped"}

# DELETE 请求 - 删除资源
DELETE https://api.example.com/comments/999
```

**请求体模板示例**：
```json
{
  "query": "{{user_query}}",
  "user_id": "{{user_id}}",
  "filters": {
    "category": "{{selected_category}}",
    "price_range": [{{min_price}}, {{max_price}}]
  },
  "pagination": {
    "page": 1,
    "limit": 10
  }
}
```

**使用场景**：
- 🔌 调用第三方 API（天气、地图、翻译）
- 🔌 与数据库系统交互
- 🔌 Webhook 回调
- 🔌 图片生成服务（Midjourney、SD 等）
- 🔌 支付接口调用
- 🔌 短信/邮件服务

**错误处理最佳实践**：
```
HTTP 请求 → 成功?
    ├─ 是 → 解析响应数据
    └─ 否 → 条件分支
            ├─ 网络错误 → 重试
            ├─ 参数错误 → 提示用户
            └─ 服务异常 → 降级处理
```

## 常见节点组合模式

### 模式 1：线性处理链

```
输入 → 处理1 → 处理2 → 处理3 → 输出
```

**特点**：按顺序处理，每个节点的输出是下一个节点的输入

**应用**：数据清洗、格式转换

---

### 模式 2：条件分支

```
        ┌─ 条件成立 ─→ 处理A
输入 ─→ 判断条件
        └─ 条件不成立 → 处理B
        ↓
    聚合 ──→ 输出
```

**特点**：根据条件选择不同的处理路径

**应用**：智能路由、优先级处理

---

### 模式 3：循环迭代

```
输入: [项1, 项2, 项3...]
    ↓
对每项执行 {
  LLM 处理
  HTTP 请求
  保存结果
}
    ↓
输出: [结果1, 结果2, 结果3...]
```

**特点**：对列表中的每个元素执行相同操作

**应用**：批量处理、并行生成

---

### 模式 4：多路并行处理

```
        ┌─ 路径A ──┐
输入 ─→ 分发       │
        ├─ 路径B ──┼─→ 聚合 ──→ 输出
        └─ 路径C ──┘
```

**特点**：多条路径并行处理，最后合并结果

**应用**：多模态处理、降级策略

---

### 模式 5：递归优化

```
初始结果
    ↓
评估质量满意?
    ├─ 是 ──→ 输出
    └─ 否 ──→ 迭代优化 ──→ (返回评估)
```

**特点**：根据反馈多轮改进

**应用**：内容优化、结果精化

## 最佳实践建议

### ✅ 节点设计原则

1. **单一职责**：每个节点完成一个明确的功能
2. **模块化**：将复杂工作流拆分为可复用的子工作流
3. **可观测性**：在关键节点添加日志和监控
4. **性能优化**：
   - 合理使用异步处理
   - 避免过深的嵌套
   - 使用缓存提高效率

### ⚠️ 常见错误

1. **过度使用代码节点**：应该先用模板转换
2. **没有错误处理**：HTTP 请求缺少异常捕获
3. **硬编码参数**：应该使用变量和输入参数
4. **工作流过长**：应该拆分为更小的工作流

### 🚀 性能优化建议

| 问题 | 解决方案 |
|------|---------|
| 工作流响应慢 | 使用异步处理、LLM 流式输出、结果缓存 |
| 知识库查询慢 | 增加相似度阈值、限制返回结果数、优化知识库索引 |
| HTTP 请求失败 | 添加重试机制、超时管理、降级处理 |
| LLM 调用费用高 | 使用更小的模型、缓存常见问题、优化提示词 |
